- **MyAnimeList OAUTH — investigated 2026-07 (diagnosis + partial fix, needs user re-test).** The browser "sign in to myanimelist.net" popup is an **HTTP Basic auth challenge** (`WWW-Authenticate: Basic realm="OAuth"`) and the `{"error":"invalid_client","message":"Client authentication failed"}` behind it are exactly what MAL's `/v1/oauth2/authorize` returns when it **doesn't recognise the `client_id`**. Reproduced against the real endpoint: any unregistered client_id (any format, with/without `redirect_uri` or `code_challenge_method`) gives that identical popup + body; a well-formed request differs only by whether the id is a real registered app. Mangarr's authorize URL is otherwise correct — all required params, valid PKCE `code_challenge` (80 hex chars). **So MAL is rejecting the Client ID value, not the request shape.** Mangarr-side fix shipped: OAuth client id/secret are now **trimmed on read** in `MalTracker`/`AniListTracker`, so a stray space/newline pasted into the field (which `Uri.EscapeDataString` was faithfully sending, e.g. `%20`) can't silently break auth — the live authorize URL now self-heals without a re-save (verified: a `"  ABCdef123456  "` id produced `client_id=ABCdef123456`). Settings help text updated to say the popup means an unrecognised Client ID. **Remaining, user-side:** if it still fails after this, the stored Client ID is wrong — re-copy the **Client ID** (not the Secret) from myanimelist.net/apiconfig into the MyAnimeList *Client ID* box and re-save. - **Recommendation engine v2 — Phase 1: semantic ("feel") matching** (2026-07). Discover now recommends by semantic similarity of descriptions, not just shared genre labels. All in-process (`Microsoft.ML.OnnxRuntime` + `Microsoft.ML.Tokenizers`, CPU; a GPU ORT build would drop in with no code change) — no Python, no training, no per-user data. **Design decisions** (ONNX in-process over Python sidecar / trained model; quantized model; CPU default): [approved plan](https://claude.ai/code/artifact/e6f24ff9-90c9-46a9-8a86-05b69bbc3e4a).
